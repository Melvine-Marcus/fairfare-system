ABSTRACT


The FairFare System is a web-based platform designed to address transparency 
and accountability challenges in the Ongata Rongai public transportation sector. 
The system provides real-time fare information and enables users to report 
transport-related incidents, addressing key challenges identified in the 
transport sector.

This report details the design, development, and implementation of the FairFare 
System. The project identifies significant gaps in fare transparency and incident 
reporting mechanisms in current transport operations. The proposed solution 
integrates user authentication, incident management, fare tracking, and 
comprehensive activity logging.

The system was built using PHP 7.4+, MySQL database, and Bootstrap 5.3.2 for the 
frontend. Key features include secure user authentication with CSRF protection, 
role-based access control, real-time incident tracking, and comprehensive admin 
dashboard with analytics.

The FairFare System successfully demonstrates the feasibility of implementing 
technology-driven solutions to improve transportation accountability. The system 
has been tested for security, functionality, and usability. Performance tests 
show the system can handle concurrent users efficiently. The implementation 
proves that such systems can enhance trust between passengers and transport 
operators while providing valuable data for service improvement.

Keywords: fare transparency, incident reporting, web application, transportation, 
accountability, Kenya

CHAPTER ONE


Introduction

The FairFare System addresses critical transparency and accountability gaps in 
public transportation services in Ongata Rongai, Kenya. Public transportation, 
particularly the matatu industry, plays a crucial role in the daily mobility of 
urban and semi-urban populations. However, passengers have historically lacked 
mechanisms to:

1. Access current fare information reliably
2. Report incidents of overcharging or misconduct
3. Track the resolution of reported issues
4. Verify accountability of service providers

This project defines "fare transparency" as the provision of clear, accessible, 
and timely information about transportation costs. "Incident reporting" refers 
to systematic mechanisms allowing users to document and communicate concerns 
about transportation services. "Accountability" in this context means ensuring 
transport operators maintain service standards and respond to passenger concerns.

The system has been developed to bridge these information gaps and create a 
structured platform for fare and incident management.


Background

Ongata Rongai is a rapidly growing town in Kajiado County, Kenya, characterized 
by:
  • Increasing population density and urbanization
  • Heavy reliance on matatu services for local transportation
  • Limited formal regulation of informal transport sector
  • Growing demand for transparency in service provision

The organizational context includes matatu operators, passengers, and 
administrative authorities with divergent interests:

Vision: To establish a transport sector characterized by transparency, safety, 
and user trust.

Mission: To provide platforms and tools enabling passengers to access fare 
information and report service issues, while enabling operators and authorities 
to monitor and improve service quality.

Business Processes Currently Undertaken:
  1. Fare Setting: Operators set fares based on routes, distances, and demand
  2. Service Delivery: Drivers provide transportation services with varying 
     quality standards
  3. Incident Handling: Currently ad-hoc, largely unregulated
  4. Complaint Resolution: Informal, undocumented processes

Current Challenges and Weaknesses:
  • No centralized fare information system
  • Lack of structured incident reporting mechanisms
  • Limited documentation of service complaints
  • No accountability mechanism for resolving issues
  • Absence of data for service quality monitoring
  • Passengers have no way to verify fair pricing
  • Operators lack data to improve service delivery
  • No mechanism for detecting systematic problems


Challenges

The FairFare System was developed to address the following specific challenges:

1. Fare Opacity: Passengers cannot easily verify if charged fares are fair and 
   compliant with current rates, leading to disputes and reduced trust.

2. Unstructured Incident Reporting: Currently, passengers report incidents 
   informally through personal contacts, with no documentation or follow-up 
   mechanisms.

3. Accountability Gaps: Without incident documentation, it is difficult to 
   identify patterns of misconduct or implement corrective measures.

4. Information Asymmetry: Operators have production data while passengers lack 
   information, creating friction and inefficiency.

5. Limited Monitoring Capacity: Authorities cannot systematically monitor 
   service quality or identify problem areas for intervention.

6. No Performance Metrics: The transport sector lacks data-driven insights for 
   improvement, perpetuating inefficient practices.

7. Trust Deficit: The lack of transparency and accountability mechanisms erodes 
   passenger confidence in the transport system.

These challenges directly impact:
  • User experience: Passengers experience uncertainty and frustration
  • Service quality: Operators lack feedback mechanisms for improvement
  • Economic efficiency: Transaction costs increase due to dispute resolution
  • Regulatory capacity: Authorities lack necessary data for oversight


Problem Statement

The Ongata Rongai transport sector lacks integrated systems for fare 
transparency and incident reporting, resulting in information asymmetry, 
accountability gaps, and reduced service quality. Passengers cannot reliably 
access current fare information or systematically report transport-related 
incidents, while operators lack structured feedback mechanisms for service 
improvement. This situation creates friction, reduces trust, and impedes 
service quality enhancement and sector growth.

The proposed system must address these gaps by providing:
  • Central repository for current fare information
  • Structured mechanisms for incident collection and tracking
  • Transparent workflow for incident resolution
  • Administrative tools for monitoring and analysis
  • User-friendly interfaces for both passengers and administrators


Proposed Solutions

The FairFare System proposes the following integrated solutions:

1. Fare Transparency Module
   Centralized database storing current fares for all routes, enabling users to 
   access fare information before traveling, reducing disputes and enabling 
   informed decision-making.

2. Incident Reporting Module
   User-friendly web interface allowing passengers to report incidents with 
   detailed information capture, including incident type, location, date/time, 
   and narrative description.

3. Incident Tracking System
   Administrative interface enabling incident status monitoring, allowing 
   reporters to track resolution progress, improving accountability and 
   transparency.

4. Analytics Dashboard
   Comprehensive dashboard providing administrators and authorities with 
   statistics on incident patterns, fare trends, user engagement, and system 
   performance metrics.

5. Activity Logging System
   Complete audit trail of system activities, enabling accountability and 
   forensic analysis of system usage.

6. User Authentication and Access Control
   Secure authentication system with role-based access control, ensuring system 
   security and appropriate privilege assignment.

7. Data Export Capabilities
   Features enabling authorized users to export data in standard formats for 
   further analysis and reporting.


Proposed System

The proposed FairFare System is a web-based platform consisting of:

Frontend Components:
  • Public interface for fare browsing and incident reporting
  • Secure login and user registration system
  • Administrative dashboard for system management
  • Responsive design supporting mobile and desktop access

Backend Components:
  • User authentication and authorization system
  • Fare management and storage system
  • Incident capture, tracking, and management system
  • Activity logging and audit system
  • Data export and reporting system
  • SMS notification system for incident updates

Database Components:
  • Users table: User accounts and credentials
  • Fares table: Current fare information by route
  • Incidents table: Reported incidents and their details
  • Activity logs table: System activity tracking
  • Fare history table: Historical fare changes for analysis

Technology Stack:
  • Backend: PHP 7.4+
  • Database: MySQL/MariaDB
  • Frontend: HTML5, CSS3, Bootstrap 5.3.2
  • Server: Apache 2.4 with mod_rewrite
  • Security: HTTPS, CSRF tokens, prepared statements

The system will be deployed on a web server accessible to users via modern web 
browsers, with responsive design supporting mobile access.


Scope

The FairFare System scope encompasses:

Included Features:
  • User registration and authentication
  • Fare information browsing and searching
  • Structured incident reporting
  • Incident status tracking and updates
  • Administrative dashboard with analytics
  • Activity logging and audit trails
  • CSV data export for incidents and fares
  • SMS notifications for incident updates
  • Role-based access control (user and admin roles)
  • CSRF protection and input validation

Excluded from Current Implementation:
  • Mobile native applications (web-responsive only)
  • Advanced predictive analytics using machine learning
  • Integration with external payment systems
  • Multi-language support (English only)
  • GPS location tracking or mapping visualization
  • Real-time communication features beyond SMS

Geographical Scope:
  • Primary focus: Ongata Rongai transport routes
  • Expandable to additional areas with database modifications

User Scope:
  • Passengers/Transport Users (regular users)
  • Transport Operators and Administrators
  • City/County Authorities (admin access)
  • System administrators

Timeline:
  • Initial deployment: December 2026
  • First iteration: 3-month period
  • Ongoing maintenance and enhancement phases

7. Contribution to Body of Knowledge:

CHAPTER TWO

Literature Review


This section reviews existing solutions addressing transportation management, 
fare transparency, and incident reporting systems. The analysis compares 3-5 
existing systems to establish the gap that the FairFare System addresses.


2.1 Existing System 1: Public Transportation Apps (E.g., Uber, Bolt)

Description:
Uber and Bolt utilize GPS technology and dynamic pricing algorithms to connect 
users with transportation services. These applications provide transparent 
pricing before ride booking, real-time tracking, and incident reporting through 
in-app frameworks.

Implementation Approach:
  • Mobile-native applications with backend APIs
  • GPS integration for Real-time locating and routing
  • Dynamic pricing algorithms based on demand
  • User rating systems for driver accountability
  • Payment integration for cashless transactions

Benefits (Advantages):
  • Real-time fare transparency and price prediction
  • Complete audit trails of all transactions
  • Accountability through rating systems and complaints tracking
  • Data-driven insights for service optimization
  • Robust incident resolution frameworks

Challenges (Disadvantages):
  • High development and infrastructure costs
  • Requires advanced technology adoption by drivers
  • Primary focus on premium users; excludes lower-income segments
  • Significant dependency on mobile penetration and data availability
  • Not suitable for informal public transportation sector in Kenya

Selling Points:
  • Market leader in urban mobility
  • Strong brand recognition
  • Seamless payment integration
  • Comprehensive data analytics


2.2 Existing System 2: JATCO (Japan Auto Transport Cooperation) Model

Description:
Traditional administrative systems for transport regulation used in many 
developed contexts. These systems maintain centralized databases of registered 
operators, licensed routes, approved fares, and complaint records. Some 
jurisdictions have begun digitizing these systems.

Implementation Approach:
  • Centralized database of licensed operators and approved fares
  • Manual incident reporting and administrative processing
  • Monthly reports compilation from operators
  • Physical or digital complaints registration
  • Periodic inspections and audits

Benefits:
  • Provides regulatory control over transport operators
  • Creates formal documentation of fares and incidents
  • Enables authorities to detect patterns and implement policies
  • Can be operated with minimal technology requirements
  • Familiar approach to regulatory authorities

Challenges:
  • Slow feedback loops - data often outdated
  • Significant administrative overhead and manual processing
  • Limited passenger access to fare information
  • Poor incident resolution mechanics
  • Not accessible to average users
  • Paper-based processes prone to loss and corruption

Selling Points:
  • Proven regulatory framework
  • Familiar to government authorities
  • Can operate with existing infrastructure
  • Formal legal standing in regulations


2.3 Existing System 3: Ushahidi Crisis Mapping Platform

Description:
Ushahidi is an open-source platform originally developed during Kenya's post-
election crisis for mapping incidents. It allows crowdsourced incident reporting 
and visualization on interactive maps. In some cities, variations have been 
adapted for transportation oversight.

Implementation Approach:
  • Crowdsourced incident reporting via web and SMS
  • Incident mapping and geospatial visualization
  • Category-based incident classification
  • Time-series analysis of incident patterns
  • Public/authenticated access with role-based controls
  • Email notification system

Benefits:
  • Open-source and customizable

Key Findings:

1. Gap in Informal Transport Sector Focus: Most existing solutions target formal 
   transportation (licensed commercial operations or premium private transport). 
   Few address the informal sector where most Ongata Rongai transport operates.

2. Fare Transparency Gap: While many systems track fares, few make this 
   information proactively available to passengers before transactions. FairFare 
   positions fare transparency as a primary feature.

3. Regulatory Integration Gap: Systems either focus on passenger convenience 
   (Uber/Bolt) or regulatory control (JATCO) but rarely integrate both 
   perspectives. FairFare bridges this gap.

4. Cost-Benefit Mismatch: Premium solutions (Uber/Bolt) are expensive to deploy; 
   manual systems (JATCO) are labor-intensive. FairFare aims for cost-effective 
   middle path.

5. SMS Integration: While some systems support SMS, few make it central to 
   operations for limited-connectivity environments.

6. Accountability Mechanism: Few systems provide transparent, user-friendly 
   incident resolution workflows visible to both reporters and operators.

7. Data Accessibility: Limited systems proactively provide operators feedback on 
   performance issues and improvement opportunities.


2.7 Contribution of FairFare System

Based on the gap analysis, FairFare System makes the following contributions:

1. Integration of Fare Transparency and Incident Reporting: Unlike most existing 
   systems, FairFare tightly integrates fare information with incident reporting, 
   recognizing that these are complementary aspects of transport accountability.

2. Informal Sector Applicability: Designed specifically for informal transport 
   sectors, balancing regulatory needs with operator sustainability concerns.

3. Low Technical Barrier to Entry: Web-based with SMS support, accessible to 
   diverse tech-literacy levels without requiring app installation.

4. Transparent Resolution Workflows: Structured incident tracking with clear 
   status updates provides accountability visible to all stakeholders.

5. Regulatory Data Provision: Automatically provides regulatory authorities with 
   structured data for oversight without adding operator burden.

6. Operator Feedback Integration: Closes loop by providing operators with 
   structured feedback on service issues.

7. Appropriate Technology Deployment: Uses proven, maintainable technologies 
   avoiding unnecessary complexity while ensuring security and reliability.

CHAPTER THREE

Methodology


This chapter describes the methodologies employed in data collection, systems 
analysis, design, testing, and implementation of the FairFare System.


3.1 Data Collection Methods

Interviews:
Structured interviews were conducted with 10 key stakeholders:
  • 4 matatu operators
  • 3 regular passengers
  • 2 transport regulatory authority representatives
  • 1 civil society organization representative focused on transport

Interview guide covered current practices, pain points, information needs, and 
preferences for system features. Interviews were conducted over 2 weeks and 
analyzed for recurring themes.

Questionnaires:
Self-administered questionnaires were distributed to 50 regular passengers on 
common Ongata Rongai routes. Questionnaire covered:
  • Current awareness of fare rates
  • Frequency of fare disputes
  • Preferred incident reporting methods
  • Technology comfort level and access

Observations:
Direct observation was conducted at 3 major transport terminals during peak 
hours (1.5 hour per terminal) to understand:
  • Current fare display and communication practices
  • Passenger-operator interactions
  • Incident manifestations and resolution attempts
  • Technology availability and usage patterns


3.2 System Analysis

Current System Analysis:
The existing informal system was documented as:
  • Fares communicated verbally or displayed on vehicle placards
  • No centralized fare information
  • Incident resolution through personal relationships or informal advocacy
  • No documentation or tracking
  • Limited accountability mechanisms

Problem identification through data analysis identified key gaps.

Functional Requirements:
  1. User registration and authentication
  2. View current fares by route
  3. Report incidents with structured information capture
  4. Track incident status
  5. Admin dashboard for incident and fare management
  6. Activity logging and audit trails
  7. Data export capabilities

Non-Functional Requirements:
  1. Security: HTTPS, CSRF protection, prepared SQL statements
  2. Performance: Page load < 2 seconds, database queries < 500ms
  3. Availability: 99% uptime
  4. Reliability: Comprehensive error handling and logging
  5. Usability: Mobile-responsive, intuitive interfaces
  6. Maintainability: Well-documented code, modular architecture
  7. Scalability: Support 1000+ concurrent users

Unnormalized Form (Initial Concept):

This design has redundancy and update anomalies.


First Normal Form (1NF):
All non-key attributes must be atomic (no repeating groups).

    Users
    ├── user_id (PK)
    ├── username (UNIQUE)
    ├── email (UNIQUE)
    ├── password
    ├── role (admin | user)
    └── is_active

    Fares
    ├── fare_id (PK)
    ├── route (string)
    ├── fare (decimal)
    ├── effective_date
    ├── created_by (FK → users.user_id)
    └── timestamp

    Incidents
    ├── incident_id (PK)
    ├── user_id (FK → users.user_id)
    ├── name
    ├── email
    ├── phone
    ├── route
    ├── incident_type
    ├── description (LONGTEXT)
    ├── status
    ├── resolved_by (FK → users.user_id)
    ├── resolved_at
    └── created_at / updated_at


Second Normal Form (2NF):
Remove partial dependencies - all non-key attributes must depend on entire 
primary key.

[For composite keys, ensure no non-key attribute depends on part of the key]

    Fare_History
    ├── history_id (PK)
    ├── route
    ├── old_fare
    ├── new_fare
    ├── changed_by (FK → users.user_id)

    ├── action
    ├── details

    ├── user_agent
    └── created_at

Tables are in 2NF as each non-key attribute depends on full primary key.


Third Normal Form (3NF):
Remove transitive dependencies - non-key attributes must depend only on the 
primary key.

Review each table:
  • Users: All attributes directly describe user identity - 3NF ✓
  • Fares: All attributes directly describe fare record - 3NF ✓
  • Incidents: All attributes describe incident report - 3NF ✓
  • Fare_History: All attributes describe historical record - 3NF ✓
  • Activity_Logs: All attributes describe activity record - 3NF ✓

No transitive dependencies exist. Schema is in 3NF.

Data Dictionary:

USERS Table:
  • user_id (INT, PK): Unique user identifier, auto-increment
  • username (VARCHAR 50, UNIQUE): User's chosen username
  • email (VARCHAR 100, UNIQUE): User's email address
  • password (VARCHAR 255): Bcrypt-hashed password
  • role (ENUM): 'user' or 'admin' role assignment
  • is_active (BOOLEAN): Whether account is active
  • phone (VARCHAR 20, NULLABLE): Optional phone number
  • created_at (TIMESTAMP): Account creation datetime
  • updated_at (TIMESTAMP): Last profile update datetime


INCIDENTS Table:
  • incident_id (INT, PK): Unique incident identifier
  • user_id (INT, FK, NULLABLE): Reporting user reference
  • name (VARCHAR 100): Incident reporter name
  • email (VARCHAR 100): Incident reporter email
  • phone (VARCHAR 20, NULLABLE): Reporter contact phone
  • route (VARCHAR 100): Affected transport route
  • incident_type (VARCHAR 50): Category (e.g., overcharging, misconduct)
  • description (LONGTEXT): Detailed incident description
  • status (ENUM): one of open|in_progress|resolved|closed
  • resolved_by (INT, FK, NULLABLE): Admin who resolved incident
  • resolved_at (TIMESTAMP, NULLABLE): Resolution completion time
  • created_at (TIMESTAMP): Initial report datetime
  • updated_at (TIMESTAMP): Last update datetime
  • Constraints: user_id and resolved_by reference users.user_id
  • Index: (user_id, created_at) for fast filtering


FARES Table:
  • fare_id (INT, PK): Unique fare entry identifier
  • route (VARCHAR 100): Route name/identifier
  • fare (DECIMAL 10,2): Fare amount in Kenyan Shillings
  • effective_date (DATE): Date fare becomes effective
  • created_by (INT, FK): User who entered the fare
  • created_at (TIMESTAMP): Record creation time
  • updated_at (TIMESTAMP): Last update time
  • Index: (route, effective_date) for quick fare lookups


FARE_HISTORY Table:
  • history_id (INT, PK): Unique history record identifier
  • route (VARCHAR 100): Route name
  • old_fare (DECIMAL 10,2, NULLABLE): Previous fare
  • new_fare (DECIMAL 10,2): New fare amount
  • changed_by (INT, FK): Admin who made change
  • changed_at (TIMESTAMP): Change datetime


ACTIVITY_LOGS Table:
  • log_id (INT, PK): Unique log entry identifier
  • user_id (INT, FK, NULLABLE): Acting user reference
  • username (VARCHAR 50): Username of actor
  • action (VARCHAR 100): Action performed (e.g., LOGIN, INCIDENT_REPORTED)
  • details (TEXT): Additional action details
  • ip_address (VARCHAR 45): Source IP address (IPv4 or IPv6)
  • user_agent (TEXT): Browser/client user agent
  • created_at (TIMESTAMP): Event timestamp
  • Index: (user_id, created_at) for audit trail queries

4.3 Security Implementation

CSRF PROTECTION:
  All forms include csrf_token hidden field generated and validated per session
  Prevents cross-site form submission attacks
  Tokens automatically regenerated on each session

RATE LIMITING:
  Login endpoint implements 5-attempt limit per 15 minutes
  Attempts tracked in session with timestamp
  Failed attempts logged for audit trail

SQL INJECTION PREVENTION:
  All database queries use PDO prepared statements
  Parameters passed as bound values, never concatenated
  Query parameters treated as string literals, not executable code

XSS PREVENTION:
  All user-supplied content passed through htmlspecialchars()
  HTML entities encoded with ENT_QUOTES flag
  UTF-8 character set enforced

PASSWORD SECURITY:
  Passwords hashed using PHP password_hash() with BCRYPT
  Hashes use automatic salt generation (cost factor: 10)
  Password verification uses timing-safe comparison to prevent timing attacks
  Minimum 8 characters required for password complexity

INPUT VALIDATION:
  Email addresses validated using filter_var() with FILTER_VALIDATE_EMAIL
  Phone numbers validated for Kenya format (07xx or +254xxx pattern)
  Text fields limited to maximum 5000 characters
  Required fields enforced on all critical forms

AUTHORIZATION:
  Role-based access control with two roles: admin and user
  Admin-only pages call require_admin() before content output
  User-protected pages call require_login() before content output
  Unauthenticated users redirected to login with return path


4.4 Database Implementation

Successfully implemented 6 normalized tables:

TABLE: users
  Stores user account information with role-based access control
  Primary Key: id
  Unique Constraints: username, email
  Indexes: email (for fast login), role (for authorization)

TABLE: incidents
  Stores all reported transport incidents
  Primary Key: id
  Foreign Keys: user_id (reporter), admin_id, resolved_by
  Indexes: route, status, created_at (for filtering and sorting)
  Soft-delete: incidents not physically deleted, marked as closed

TABLE: fares
  Stores current and historical route-specific fares
  Primary Key: id
  Foreign Key: created_by (admin who created)
  Unique Constraint: route + effective_date combination
  Indexes: route, effective_date

TABLE: fare_history
  Maintains complete audit trail of fare changes
  Primary Key: id
  Foreign Keys: fare_id, changed_by (admin)
  Supports trend analysis and dispute resolution

TABLE: activity_logs
  Tracks all user actions for audit compliance
  Primary Key: id
  Foreign Key: user_id
  Indexes: user_id, timestamp (for query performance)

TABLE: admin_logs
  Tracks administrative actions separately
  Primary Key: id
  Foreign Key: user_id (admin)
  Indexes: timestamp (for chronological queries)
